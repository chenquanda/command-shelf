# CommandShelf（高性能常用命令看板）

> 本文由想法仓库迁移而来，记录产品方向与长期边界；第一版产品与技术方案已经确认，当前进入正式实现阶段。

## 想法概述

做一个面向个人使用的高性能常用命令看板，用来沉淀自己平时真正使用但又容易忘记的 Linux、Docker、Git 等命令。

界面大致分为左右两部分：左侧是数量有限且位置稳定的分类菜单，右侧展示当前分类中的全部常用命令。用户需要某条命令时，主动进入自己判断的分类并浏览少量内容，很快找到目标命令。

这个工具不追求收录大量命令，也不试图通过搜索或 AI 替用户直接给出答案。它更像是个人长期维护的“命令记忆板”：通过主动查找、反复看到和逐渐熟悉固定位置，帮助用户真正记住常用命令。

## 要解决的问题

开发者经常会用到 Linux、Docker、Git、SSH、网络诊断等命令，但很多命令并没有高频到可以完整记住。常见情况包括：

- 知道应该使用哪类命令，但记不清参数、选项顺序或完整写法。
- 命令曾经记录在 Markdown、文本文件或聊天记录中，真正需要时还要打开文件并翻找。
- 使用搜索引擎或询问 AI 虽然可以获得答案，但操作路径偏长，而且每次都由外部工具直接给出结果，不利于形成自己的长期记忆。
- 通用命令手册内容太多，个人真正反复使用的命令只占很小一部分，过多信息反而增加查找成本。
- 现有笔记工具通常比较重，启动、导航和编辑过程不够直接，不适合临时查询一条命令。

## 核心定位

这是一个本地优先、内容克制、打开即看的个人常用命令看板，而不是：

- 完整的 Linux、Docker 或 Git 命令百科；
- 依赖关键词或语义检索的命令搜索工具；
- 通过 AI 临时生成命令的问答工具；
- 在线文档聚合器；
- 直接执行命令或管理远程主机的终端工具。

工具的核心价值不是“知道所有命令”，而是让用户积累、整理并反复接触自己真正用过的命令。

## 关键产品原则

### 少而常用

每个分类只保存个人确实使用过、预计还会再次使用的命令。内容数量保持在能够快速扫完的范围内，不为了显得完整而收录大量低价值命令。

手动筛选不是产品能力不足，而是这个工具帮助用户形成个人知识资产的重要过程。

### 主动查找

用户先回忆命令属于哪个分类，再主动进入分类浏览。这个过程保留了一定的记忆努力，有助于反复建立“使用场景—分类—命令”的关联。

因此当前明确不提供搜索功能，也不通过 AI、语义匹配或自动推荐绕过主动查找过程。

### 位置稳定

分类和命令默认保持固定顺序，并允许用户手动排序。系统不因使用频率、最近访问时间或推荐算法自动改变位置，避免破坏用户逐渐形成的空间记忆。

### 快速直达

应用应当打开速度快、界面层级少。进入分类后直接看到全部命令，不使用多层目录、分页或复杂详情跳转。

### 本地可控

命令数据由用户自己添加、修改和删除，并始终先保存在本地。浏览、编辑和复制不依赖网络；需要在两台电脑之间迁移数据时，可以通过用户自己的 GitHub 私有仓库显式拉取或推送。

应用不内置 GitHub 账号、Token 或云端服务，认证由系统 Git 及其现有凭据管理。即使同步暂时不可用，本地数据和离线核心能力也不能受影响。

## 典型使用场景

### 临时回忆命令

用户准备查看 Docker 容器最近一段日志，只记得大概需要使用 `docker logs`。打开看板，进入“Docker”分类，扫一眼就能找到带有常用参数的完整命令并复制。

### 记录刚解决的问题

用户刚刚通过资料查到一个以后可能再次使用的 Linux 命令。问题解决后打开看板，在“Linux”或更具体的分类中新增该命令，并写下一句用途说明，避免下次重新查资料。

### 通过重复浏览形成记忆

用户多次进入同一分类寻找命令。由于分类和命令位置稳定，逐渐能够记住命令写法，或者至少记得它在看板中的位置。

### 保存个人化写法

通用资料提供的命令通常需要替换路径、端口、分支名或容器名。用户可以保存符合自己习惯的命令模板、常用参数组合和简短提醒。

### 在两台电脑之间同步

用户在电脑 A 使用前拉取最新数据，完成修改后推送；切换到电脑 B 时再次拉取，继续使用或修改后推送。同步遵循“拉取 → 修改 → 推送”的串行使用方式，不建设多人协作和冲突处理流程。

## 界面构想

### 左侧分类菜单

左侧展示所有命令分类，例如：

- Linux
- Docker
- Git
- SSH
- 网络诊断
- 文件处理
- 数据库
- 自定义分类

分类数量应当有限，支持新增、重命名、删除和手动排序。分类可以显示命令数量，但不需要复杂统计信息。

### 右侧命令内容区

右侧一次展示当前分类中的全部命令。整体适合快速纵向浏览，可采用紧凑列表或轻量卡片形式。

每条命令可以包含：

- 简短标题：用任务或目的描述，例如“查看容器最近 100 行日志”；
- 命令正文：保留完整可复制的命令；
- 用途说明：说明这条命令解决什么问题；
- 参数提示：说明需要替换的路径、端口、分支名、容器名等内容；
- 示例或常用变体：保存个人经常使用的参数组合；
- 注意事项：标记删除、覆盖、强制操作等潜在风险；
- 个人备注：记录容易忘记的前提、限制或使用经验。

内容默认直接可见，避免每条命令都必须进入详情页才能查看。说明较多时可以折叠次要内容，但命令标题和命令正文应始终容易看到。

## 核心能力

### 分类管理

- 新增自定义分类；
- 修改分类名称；
- 删除不再需要的分类；
- 手动调整分类顺序；
- 在分类之间移动命令。

### 命令管理

- 添加新的命令；
- 编辑已有命令及其说明；
- 删除不再使用的命令；
- 复制命令正文；
- 手动调整命令在分类中的顺序；
- 将多种相近写法作为同一用途下的变体组织在一起。

新增命令时应尽量简单。最少只需填写标题和命令正文，其余说明均可按需补充，避免记录成本过高导致用户不愿维护。

### 本地数据维护

- 自动保存用户修改；
- 支持导出全部数据作为备份；
- 支持从备份恢复或迁移数据；
- 使用单个、格式稳定且便于版本管理的 `commands.json` 保存分类与命令数据；
- 允许用户选择一个已经克隆到本机的独立数据仓库；
- 提供显式“拉取”和“推送”操作，并显示已同步、本地有修改、同步中和同步失败状态；
- 拉取成功后校验并重新加载命令数据，推送时自动完成保存、暂存、提交和远端推送；
- 不自动强制推送，不在应用内解决 Git 冲突，同步失败时必须保留本地数据；
- 数据格式保持透明、稳定，避免用户被特定版本或平台锁定。

## 性能与体验要求

性能不是普通的技术优化项，而是这个想法成立的前提。用户只是临时查看一条命令，如果工具本身打开缓慢或长期消耗大量资源，就会重新选择文本文件、浏览器或 AI。

期望具备以下体验：

- 冷启动足够快，打开后立即可以浏览内容；
- 常驻时占用较少内存，不常驻时能够完全退出；
- 空闲状态几乎不持续占用 CPU；
- 切换分类和复制命令没有可感知等待；
- 不因命令数量逐渐增加而出现明显卡顿；
- 不为了视觉效果引入重型动画、复杂渲染或大量后台任务；
- 离线状态下浏览、编辑、排序、保存和复制等核心能力正常可用，只有 GitHub 同步需要网络。

具体的性能指标和实现技术暂不在想法阶段确定，但未来选择产品形态与技术方案时，应把启动速度、内存占用和空闲 CPU 占用作为重要判断依据。

## 内容组织规则

为了让看板保持“看一眼就能找到”的特性，可以遵循以下规则：

- 分类按用户理解的工具或使用场景组织，不追求统一的理论分类法；
- 标题优先描述“要做什么”，而不是只写命令名称；
- 同一用途的相近命令尽量放在一起，避免散落成多个难以区分的条目；
- 命令过时或已经记牢且不再需要查看时，可以归档或删除；
- 某个分类变得过长时，优先整理、合并或拆分类，而不是引入搜索；
- 危险命令必须有醒目的风险提示，避免复制后误操作；
- 命令排序由用户决定，并长期保持稳定。

## 明确不考虑的能力

当前方向明确不考虑：

- AI 生成、解释或推荐命令；
- 语义搜索、关键词搜索或全文检索；
- 自动从互联网抓取命令；
- 建设覆盖所有工具的完整命令库；
- 根据使用频率自动排序；
- 信息流、社区分享或公开命令市场；
- 内置 GitHub 登录、Token 管理、自动冲突合并或多人协作；
- 在看板中直接执行命令。

这些限制共同保证产品始终围绕“个人积累、主动翻看、反复记忆”展开。未来即使增加其他能力，也不应破坏这一核心使用方式。

## 可以继续扩展的方向

在不改变核心定位的前提下，未来可以继续讨论：

- 通过全局快捷键快速打开或隐藏看板；
- 支持窗口置顶、托盘入口或小尺寸侧边栏形态；
- 为命令中的可替换参数提供轻量占位提示；
- 支持浅色、深色和高密度展示模式；
- 支持归档已经记熟但暂时不想删除的命令；
- 支持多个本地命令集合，例如工作、个人服务器或不同技术栈；
- 支持导入、导出便于人工阅读和版本管理的文本格式。

## 已确认的第一版边界

- 使用普通 Windows 桌面窗口并交付 EXE；快捷唤起、常驻侧边栏和跨平台支持进入后续迭代。
- 同步入口固定提供“拉取”和“推送”，不自动访问网络，不在应用内登录 GitHub 或解决冲突。
- 同步状态包括未配置、已同步、本地有修改、同步中和同步失败；同步中暂停数据修改，但浏览和复制继续可用。
- 原型命令仅用于展示，正式用户数据从空内容开始，避免把通用样例混入个人命令资产。
- 归档、多个命令集合、分类高级管理和主题模式不进入第一版。

## 已确认的技术方向

- 第一版使用 Tauri 2、Rust 与系统 WebView2，直接复用已确认的 HTML/CSS/JavaScript 界面，首发 Windows x64 NSIS 安装包。
- `commands.json` 从第一版固定加入整数 `schemaVersion: 1`；分类和命令数组顺序就是手动顺序，稳定 ID 不因重排变化。
- 本地保存采用校验、应用外备份和 Windows 原子替换；磁盘基线被外部修改时停止覆盖并提示重新加载。
- Git 同步集中在 Rust 后端，以固定参数数组非交互调用系统 Git，只允许快进拉取、普通推送和仅暂存 `commands.json`。
- 技术壳实验已经验证真实原型启动、文件往返、系统 Git 与 NSIS 打包；正式版本继续用既定启动、内存、CPU 和体积门槛验收。
