## 项目定义

- 核心问题：开发者经常使用 Linux、Docker、Git 等命令，但容易忘记完整写法和输出含义；从零搜索或询问 AI 虽然直接，却不利于积累个人命令资产和形成记忆。
- 目标用户：需要频繁查阅少量常用命令的个人开发者。
- 项目目标：提供打开快、资源占用低、位置稳定的个人命令看板，让用户主动按分类翻看、复制并逐渐记住常用命令。
- 成功结果：用户打开工具后无需搜索即可在少量分类和命令中快速定位目标，并能在执行前通过参考输出理解结果形态。
- 第一版范围：分类浏览、命令浏览、参考输出、新增和编辑命令、复制命令、手动排序、本地 JSON 保存、GitHub 拉取与推送。
- 明确不做：AI 生成或解释、关键词或语义搜索、在线命令百科、自动推荐和排序、直接执行命令、内置 GitHub 登录、Token 管理、冲突解决和多人协作。
- 核心流程：打开看板 → 主动选择分类 → 浏览少量命令 → 查看参考输出 → 复制命令；或拉取远端数据 → 新增或修改命令 → 推送到个人数据仓库 → 在另一台电脑拉取。

### 核心流程

1. 首次连接：用户在系统中克隆个人数据仓库 → 打开应用 → 选择本地仓库路径 → 应用校验并加载 `commands.json` → 显示“已同步”。
2. 浏览复制：选择分类 → 浏览命令 → 按需展开详情 → 复制命令或参考输出；该流程离线可用。
3. 修改推送：新增、编辑或排序 → 自动保存本地文件 → 状态变为“本地有修改” → 点击“推送” → 成功后回到“已同步”。
4. 跨电脑使用：电脑 A 推送完成 → 电脑 B 打开后点击“拉取” → 校验并加载最新数据 → 继续浏览或修改。
5. 安全失败：任何同步错误 → 立即停止远端操作 → 保持本地文件 → 显示失败原因和可执行建议 → 用户修复后重试。

| 同步失败 | 显示失败原因和本地数据未受影响 | 根据错误重试拉取或推送、继续离线浏览与编辑 |
| 空分类 | 展示空状态和新增命令入口 | 新增命令、切换分类 |

### 边界与异常规则

- 第一版采用普通 Windows 桌面窗口；全局快捷键、小窗唤起、侧边栏形态和跨平台支持进入后续范围。
- 分类和命令顺序只因用户显式操作变化，不根据访问频率或最近使用自动调整。
- 新建的正式数据文件默认为空；原型中的命令仅用于展示和验收，不自动写入用户数据仓库。
- 本地保存成功后才显示“本地有修改”；同步状态不能代替本地持久化结果。
- 有本地修改时禁止更换仓库；同步进行中禁止新增、编辑和排序，避免数据基线变化。
- 拉取不会覆盖未推送的本地修改；推送禁止强制覆盖；冲突或非预期分叉统一停止并交由仓库外处理。
- 仓库路径是当前电脑配置，不进入 `commands.json`；两台电脑各自选择本地克隆路径。
- 应用不定时拉取、不后台推送，也不因启动或退出自动访问网络。
- 参考输出始终提示“实际内容因环境不同而变化”。

## 技术实现方案

### 1. 技术栈决定

- 桌面壳采用 **Tauri 2 + Rust**，前端继续使用已经验收的原生 HTML、CSS 与 JavaScript；首发目标固定为 Windows x64。
- Windows 使用系统 WebView2 Evergreen Runtime 渲染界面；发布为 NSIS 每用户安装包，采用小体积 bootstrapper 模式，在机器缺少 WebView2 时再联网安装运行时。
- 后端只暴露窄接口的 Tauri Commands，不启用通用 Shell 能力。Git 必须由 Rust 使用固定程序名和参数数组调用，前端不能提交任意可执行文件或 Shell 字符串。
- 第一版不引入 Node 运行时、前端框架、数据库、服务端或 GitHub API；构建依赖只存在于开发机，发布包不携带 Rust、Node 或 Git。

选择依据：Tauri 可以直接复用已通过体验与可访问性验收的页面，当前 Windows 开发环境也已具备 Rust、MSVC 与 WebView2。Wails 同样依赖 WebView2，无法改变主要运行时内存来源，且当前本机 Go 工具链为 `windows/386`，与其 Windows AMD64/ARM64 支持目标不一致；Electron 需要随应用携带 Chromium；纯原生 UI 虽有进一步降低内存的潜力，但会重写已确认界面并增加可访问性与视觉回归风险。只有正式版本越过性能关口时才重新评估纯原生界面。

### 3. 组件与信任边界

| 组件 | 职责 | 不负责 |
|---|---|---|
| `WebUi` | 分类与命令展示、表单、可访问交互、同步状态呈现 | 不直接访问文件、Git 或网络 |
| `AppService` | 串行编排加载、保存、拉取、推送，向前端返回统一快照 | 不拼接 Git 命令文本 |
| `ConfigStore` | 在 `%APPDATA%\CommandShelf\config.json` 保存仓库路径与窗口配置 | 不写入数据仓库 |
| `CommandStore` | 校验、读取、序列化、备份和原子替换 `commands.json` | 不承担远端同步 |
| `GitSync` | 校验仓库、检查基线、抓取、快进、提交和推送 | 不执行强制推送、冲突合并或凭据管理 |
| `ProcessRunner` | 以无窗口、非交互、超时方式执行固定 `git` 子命令 | 不接受前端提供的程序名或自由参数列表 |

前端可调用的后端接口固定为：`load_app`、`choose_repository`、`save_document`、`pull_repository` 和 `push_repository`。每次返回完整 `AppSnapshot`，其中包含有效文档、同步状态、仓库显示名、最近同步时间和可操作错误；Rust 后端是持久化与同步状态的单一事实来源。

### 4. 数据格式与配置

`commands.json` 使用 UTF-8、两空格缩进和 LF，数组顺序就是用户手动顺序，不增加容易失配的 `order` 字段。稳定 ID 使用应用生成的 UUID 字符串，重排时不改变 ID。

```json
{
  "schemaVersion": 1,
  "categories": [
    {
      "id": "分类 UUID",
      "name": "Linux",
      "commands": [
        {
          "id": "命令 UUID",
          "title": "查看当前运行进程",
          "command": "ps aux",
          "description": "查看系统当前全部进程。",
          "usage": "ps aux",
          "parameters": [
            { "name": "a", "description": "显示所有终端上的进程" }
          ],
          "outputExample": "USER  PID  %CPU  %MEM  COMMAND",
          "riskNote": "",
          "notes": ""
        }
      ]
    }
  ]
}
```

校验规则：`schemaVersion` 必须等于 `1`；分类 ID、分类名、命令 ID、标题、命令正文和参考输出不能为空；全部 ID 在各自文档中唯一；参数名与说明成对出现；文件最大 10 MB；拒绝未知顶层版本但保留未知的可选业务字段需要等到未来迁移方案明确后再支持。正式初始化内容为 `{"schemaVersion":1,"categories":[]}`。

机器配置使用独立的 `configVersion: 1`，保存规范化仓库根路径、窗口大小与最近有效文档哈希。配置和备份均不进入 Git。选择仓库时必须验证目录存在、目录本身是 Git 根目录、存在 `origin` 和当前分支上游；仓库缺少 `commands.json` 时，经用户选择连接后创建空文档并进入“本地有修改”，不自动产生远端提交。

### 5. 本地加载与保存协议

1. 启动时读取机器配置；未配置或路径失效则返回“未选择仓库”，不访问网络。
2. 加载 `commands.json` 前完成大小、UTF-8、JSON 结构和业务字段校验；失败时保留内存中的上一份有效快照并返回 `DATA_INVALID`。
3. 编辑请求携带前端基于的文档哈希；后端先比较磁盘哈希，若文件已被外部修改则返回 `BASELINE_CHANGED`，避免静默覆盖。
4. 保存时先完整校验，在仓库同目录写临时文件并刷新到磁盘；旧文件复制到 `%APPDATA%\CommandShelf\backups\<仓库指纹>`，最多保留最近 10 份。
5. Windows 上通过带替换与写穿语义的文件 API 原子替换目标文件；任何一步失败都不删除最后一份有效文件。成功后更新哈希，界面状态才变为“本地有修改”。
6. 单进程写锁串行化保存和同步；同步中后端再次拒绝写入，不能只依赖界面按钮禁用。

### 6. Git 拉取协议

1. 获取全局同步互斥锁并验证仓库、系统 Git、`origin` 与上游分支；子进程最长运行 60 秒，超时后终止。
2. 检查整个工作区与暂存区。存在任何未提交变化或本地未推送提交时停止，返回“请先推送本地修改”；第一版专用数据仓库不允许夹带其他待处理文件。
3. 使用非交互环境执行 `git fetch --prune origin`，不弹出终端窗口；认证不可用时给出在系统终端完成 Git 登录或凭据配置的建议。
4. 只接受当前提交可快进到上游的关系；已分叉返回 `GIT_DIVERGED`，不做 rebase、merge、reset 或 force。
5. 在改变工作区前用 `git show <upstream>:commands.json` 读取并校验远端候选数据；缺失或格式无效时停止，当前文件和界面均不改变。
6. 执行 `git merge --ff-only <upstream>`，重新校验磁盘文件后切换界面快照，记录最近同步时间；失败仍显示上一份有效内存数据。

### 7. Git 推送协议

1. 先确认最近编辑已通过本地保存，并重新校验 `commands.json` 与基线哈希。
2. 拒绝已有暂存内容、`commands.json` 之外的工作区变化或不支持的仓库状态，避免应用提交用户的其他文件。
3. 执行 `git fetch --prune origin`，确认远端上游是本地 `HEAD` 的祖先；远端领先或已分叉时停止，不创建新提交，并提示在仓库外处理。
4. 仅执行 `git add -- commands.json`；若存在暂存差异，提交信息固定为 `chore(data): sync CommandShelf data <本地时间>`，提交身份使用用户现有 Git 配置。
5. 执行普通 `git push`，永不附加 `--force`。推送失败时保留工作文件和已创建的本地提交，状态仍为“本地有修改”，用户可在修复网络或认证后重试。
6. 推送成功后重新计算工作区与上游关系；只有工作区干净且本地不领先时才进入“已同步”。

### 8. 状态、错误与运行约束

- “未选择仓库”由配置缺失或路径失效得出；“同步中”由互斥任务得出；“本地有修改”由 `commands.json` 工作区变化或本地领先提交得出；其余正常状态为“已同步”。最后一次错误单独附着，不会伪装成本地数据丢失。
- 错误返回固定 `code`、用户说明、下一步动作与是否可重试。首版至少覆盖 `REPO_NOT_CONFIGURED`、`GIT_NOT_FOUND`、`GIT_AUTH_REQUIRED`、`WORKTREE_DIRTY`、`REMOTE_AHEAD`、`GIT_DIVERGED`、`DATA_INVALID`、`BASELINE_CHANGED`、`WRITE_FAILED`、`GIT_TIMEOUT` 和 `GIT_FAILED`。
- Git 子进程使用 `std::process::Command` 的参数数组、规范化 `current_dir`、`CREATE_NO_WINDOW`、`GIT_TERMINAL_PROMPT=0` 与有限输出缓冲；不经过 `cmd.exe` 或 PowerShell。
- 所有界面资源随应用打包，CSP 默认只允许本地资源；应用本身不发 HTTP 请求。日志只记录操作类型、阶段、耗时、错误码和 Git 退出码，不记录命令正文、参考输出或凭据内容。
- 正式实现必须为数据校验、外部基线保护、原子写入、Git 状态解析和错误映射提供 Rust 单元测试；用临时裸仓库和两个本地克隆完成拉取、推送、远端领先、分叉与失败恢复集成测试。
